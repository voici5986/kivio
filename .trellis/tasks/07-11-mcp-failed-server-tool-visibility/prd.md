# PRD: MCP 失败服务器工具降级可见

## Goal

MCP server 连接失败时,工具不再从聚合工具列表中"消失"——有没有是一回事,能不能用是另一回事。对齐 Claude Code v2.1.205+ / VS Code / Codex 的做法:工具列表保留(缓存),连接状态单独呈现,并让模型知道失败事实。

## 背景

当前连接失败 → 该 server 的工具完全不进列表 → 模型回答"没有这个工具",用户无法区分"没配置"和"配置了但坏了"。Kivio 的调用路径 `mcp_call_tool` 走 get-or-connect,天然支持"列表里有 → 调用时自动重连",保留工具列表后 transient 失败可自愈。

## Requirements

1. **曾成功连接过的 server**:握手失败/掉线后,其上次成功拉取的工具列表仍进聚合工具列表(模型可见、可调用)。调用时 get-or-connect 自动重连;重连仍失败则把连接错误作为工具执行错误(is_error)返回给模型,模型据此告知用户"server 挂了"而非"工具不存在"。
2. **从未成功连接过的 server**(无缓存工具):无法凭空造工具定义,在系统提示词工具段落注入一行"server X 已配置但当前连接失败",让模型知道其存在。
3. UI 不变:`mcp-server-state` Error 事件与设置页 Error 显示已有,复用。

## Acceptance Criteria

- [ ] server 成功连接一次后使其不可用(kill 进程/坏 endpoint),下次列工具:该 server 的工具仍在聚合列表中。
- [ ] 调用该工具:server 可重连则正常执行;不可达则返回含连接错误的工具错误,不 panic、不阻塞其他工具。
- [ ] 从未连上的 server:列表无其工具,但系统提示词含"配置但连接失败"说明。
- [ ] 聚合列表缓存策略重新评估:降级列表不得掩盖 server 恢复(恢复后下次列取应看到新工具表)。
- [ ] 现有 mcp:: 测试(51 个)保持通过,新行为有回归测试。

## Notes / 非目标

- 不做 HTTP GET SSE 通知流(list_changed for HTTP)。
- 不做 UI 改动。
